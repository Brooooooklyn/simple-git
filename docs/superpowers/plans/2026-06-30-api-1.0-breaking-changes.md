# Plan: API 1.0 breaking changes

Branch: `feat/api-1.0-breaking-sweep` (base `4cdc016`, off `main`). PR #143.

The pre-1.0 API freeze shipped on this branch in two phases, both **breaking on
purpose** — backward compatibility is not a goal. This document merges the two
phase plans that were executed in order:

- **Phase 1 — Breaking sweep** (started 2026-06-30, 10 tasks): bitflags → raw
  `number`, times → JS `Date`, counts/sizes → `number` (no bigint), config
  getter/setter redesign, `find*` nullability + naming/dedup, callback shapes →
  objects, additive async network I/O, doc fixes.
- **Phase 2 — Polish** (started 2026-07-01, 8 tasks, layered on Phase 1):
  `error.code` token union (sync + async), `Repository.dispose()`/`free()` with
  machine-enforced liveness, Windows fetch-EBUSY test fix, remaining renames +
  `Buffer` widening, `updateTips` used-callbacks guard, file-date `Date | null`,
  documentation.

Each phase keeps its own Global Constraints and task list below, verbatim from
its original plan.

# Phase 1 — API 1.0 breaking sweep

Branch: `feat/api-1.0-breaking-sweep` (base `4cdc016`, off `main`).

Goal: freeze a coherent, consistent public API before tagging 1.0. Source-verified
findings from a 3-reviewer audit (git2 0.21 / libgit2 1.9.4). Four conventions the
user locked in: **time → JS `Date`**, **bitflags → raw `number` bits**,
**counts/sizes → `number` (no bigint)**, **add async network I/O (additive)**.

This is a BREAKING release on purpose. Backward compatibility is NOT a goal.

## Global Constraints (bind every task)

- napi-rs v3 over `git2` 0.21. Errors go through `IntoNapiError` (`.convert(msg)` /
  `.convert_without_message()`); never `unwrap`/`expect`/panic on a fallible path.
- **Bitflag surfaces use raw `number`**, OR-able by callers. Keep the named
  `const enum`s but their discriminants MUST equal the real libgit2 bit values.
  Rust input conversion uses `git2::X::from_bits_truncate(n)`; returned bitsets
  use `.bits()`. No single-variant `match` with a lossy `_ =>` arm.
- **All time-bearing public APIs return/accept JS `Date`** (via `chrono::DateTime<Utc>`),
  matching `Commit.time()`. No bare seconds/ms numbers for instants.
- **All counts/sizes are `number`**, not `bigint`. Counts cast `as u32`; byte sizes
  use `i64`/`f64` so they map to `number` (lossless to 2^53 = 9 PB).
- Generated `index.js` / `index.d.ts`: after `yarn build:debug`, commit the REAL
  napi output (`git add index.js index.d.ts`). NEVER hand-edit; never `git checkout`
  them. (See memory `napi-build-loader-drift`.)
- Per-task verify loop (all must pass before reporting DONE):
  `cargo fmt -- --check` · `cargo clippy --all-targets` (deny warnings) ·
  `yarn build:debug` · `yarn test`.
- Tests: ava `.mjs` in `__test__/`. Mutating tests use `os.tmpdir()` repos with
  `git config commit.gpgsign false` AND `git config core.autocrlf false`
  (Windows-safe). NEVER use a caret `^` in an `execSync` git command (cmd.exe eats
  it) — use `git rev-list --parents -n 1 <oid>` etc. Read file content with a
  CRLF-tolerant compare when relevant. (See memory `simple-git-panic-and-errorcode-gotchas`.)
- Update `README.md` for any API whose name/shape/type changes, in the same task.
- Commit messages end with: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.

## Chosen order & rationale (READ FIRST)

Order: **T1 primitives → T2 atomic error flip → T3 dispose Rust API → T4 dispose/fetch test edits → T5 API renames + Buffer → T6 updateTips → T7 file-date `Date|null` → T8 docs (last)**.

Land the cross-cutting error-code change (T1-T2) FIRST, before dispose and the other repo.rs/remote.rs edits. Evidence-backed justification:

- `IntoNapiError::convert` flips from `napi::Error<Status>` to `crate::Result<T>` (`napi::Error<GitCode>`). This is an **atomic big-bang**: the instant the trait impl flips, every `.convert()?` sync site AND every async `compute()` body stops compiling until all are fixed together (the flip is one trait `impl`, and each file's bare `Result<T>` fns are flipped wholesale by a `use crate::Result;` shadow — you cannot flip "one fn"). So it must land as one indivisible task on a clean tree; it cannot be interleaved with dispose.
- If dispose landed first, its accessor would return `napi::Result<&Repository>` and its ~62 `self.inner()?` edits would be authored in the `Status` universe. T2's flip would then **break every `self.inner()?`** in the now-`crate::Result` methods (the `?` needs `From<Error<Status>> for Error<GitCode>`, which is orphan-blocked and does not exist), forcing a re-edit of the accessor + all closure `inner()` sites. Landing T2 first makes the whole crate `crate::Result` so dispose authors its edits **once** in final form.
- The two small breaking guards that add `Error::new` (updateTips → `GitCode::InvalidArg`) land AFTER T2 so they are written with the final token directly (no re-token pass).
- File-date `Date|null` lands after T2 **and** T3 so it rebases onto the final task `run()`/`reject()` shapes and the final `self.inner()?` accessor, and owns the one-time deletion of the now-dead `NotNullError`.
- Docs land **last**, after the file-date behavior change, so the "returns null" tooltips match the shipped `Date | null` signatures and the regenerated index.d.ts diff is doc-comment-only.
